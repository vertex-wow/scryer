# Milestone 13 — API Stub Autogeneration

## Goal

`dev/gen-api-stubs.ts` — a dev-only script that extracts `Interface/AddOns/Blizzard_APIDocumentationGenerated/*.lua` from the WoW game installation, parses every function and table definition, and auto-generates TypeScript stub files under `src/lua/api-stubs/`. After this milestone, the C\_\* sandbox coverage is exhaustive and self-maintaining — no more hand-adding stub entries when Blizzard adds or changes API functions.

## Directory layout

```
src/lua/
  api-stubs/
    retail/                 ← generated from retail run (canonical/full API set)
      C_Texture.ts
      C_Covenants.ts
      ...
    classic/              ← delta only: namespaces absent from or differing from base
      ...
    classic_era/          ← delta only: same pattern
      ...
    manifest.retail.ts    ← AUTO-GEN: functions present in retail
    manifest.classic.ts   ← AUTO-GEN: functions present in classic
    manifest.classic_era.ts
    index.ts              ← registerStubs(lua, flavor)

  api/
    retail/                 ← manual overrides, mirrors api-stubs/retail/ layout
      C_Texture.ts        ← imports type from api-stubs/retail/C_Texture.js (orphan guard)
      ...
    classic/              ← manual overrides for classic-specific behavior
      ...
    index.ts              ← registerOverrides(lua, flavor)

  wow-api.ts              ← wire: registerStubs → registerOverrides → inline stubs (C_Timer etc.)
```

## File naming

**Namespace-based.** Stub file name = the Lua namespace identifier (`C_Texture.ts`, `C_AuctionHouse.ts`).

WoW's own `/api` documentation uses the section Name field (e.g. "TextureUtils") rather than the namespace, which is confusing — we use the namespace since that is what Lua code actually calls. Namespace names are Lua identifiers (alphanumeric + underscore) — always valid filenames on all platforms.

**Global functions** (no namespace): fall back to the doc table's `Name` field, prefixed with `_` (`_Base.ts`, etc.).

**Multi-namespace source files**: one source `.lua` with multiple `AddDocumentationTable` calls produces one output `.ts` per namespace. Clean 1:1 namespace → file mapping.

## Stub format

Each stub file exports a **Lua code string**, not TypeScript functions. TypeScript functions were rejected because wasmoon converts JS objects/arrays to `userdata` — `pairs()` on userdata crashes the Lua VM. Lua `_tbl`/`_nil` local helpers (defined in the registration prelude) return proper Lua tables.

```typescript
// AUTO-GENERATED by dev/gen-api-stubs.ts — DO NOT EDIT
// @gen-hash: sha256:abc123...
// @sig-hash: sha256:def456
// Source: Blizzard_APIDocumentationGenerated/textureutilsdocumentation.lua
export const C_Texture = `
if C_Texture == nil then C_Texture = {} end
C_Texture.GetAtlasElements = _tbl
C_Texture.GetAtlasInfo = _tbl
C_Texture.GetAtlasExists = _nil
-- Events: TEXTURE_ATLASES_UPDATED
`;
```

Two hashes in the header:

- `@gen-hash` — sha256 of the source `.lua` bytes; skip rewrite if unchanged
- `@sig-hash` — sha256 of the function-name:return-type pairs; used for classic delta detection

Zero-value rules (applied to Returns definitions from the game docs):

- `Type = "table"` → `_tbl` (game uses this for both typed arrays and untyped tables; `InnerType` identifies the element type but doesn't change the stub)
- `Nilable = false` AND type name is a known Structure → `_tbl`
- `IsArray = true` → `_tbl` (legacy compat; not actually present in retail game docs)
- Everything else → `_nil` (Lua nil via empty function)

Scalar values (numbers, booleans) are not fabricated — returning nil surfaces bugs rather than masking them.

Events emitted as trailing Lua comment only. Tables/Structures consumed only for zero-value decisions. CallbackType and Enumeration entries ignored.

## Base vs delta directories

- **`retail/`** — generated from a retail extraction. Full retail API surface.
- **`classic/`** — generated from a classic extraction. Only namespaces that are absent from `retail/` or present with a different return shape.
- **`classic_era/`** — same pattern vs `retail/`.

Parameter differences between flavors are ignored (stubs use `unknown` for all params; Lua doesn't enforce arity). Only return shape differences matter for the delta.

**Stubs are never deleted.** Runs accumulate. The manifest controls what actually gets registered at runtime.

Given Blizzard's conservative development culture (Lua 5.1 since ~WoW v2, no major API rewrites between expansions), the vast majority of classic/classic_era APIs are identical to retail. Delta directories are expected to be small.

## Manifests

One manifest file per flavor, always fully regenerated, lists exactly which functions exist in that flavor:

```typescript
// AUTO-GENERATED by dev/gen-api-stubs.ts — DO NOT EDIT
export const classicManifest = {
  C_Texture: ["GetAtlasElements", "GetAtlasInfo", "GetAtlasExists"],
  C_AddOns: ["IsAddOnLoaded", "LoadAddOn"],
  // ...
} as const;
```

Manifests drive registration — only functions listed for the active flavor get registered. Stub files are the union of all flavors; the manifest is the filter.

## Registration

`registerStubs(lua, flavor)` in the generated `index.ts` runs 1–2 `doString` calls:

```typescript
// Generated index.ts (simplified)
const _pre = "local _nil = function() end\nlocal _tbl = function() return {} end\n";
const _retailLua = _pre + [C_Texture, C_Map /* ~580 namespaces */].join("\n");

export async function registerStubs(lua, flavor) {
  await lua.doString(_retailLua); // retail: one call
  if (flavor === "classic") await lua.doString(_classicLua); // delta on top
}
```

`registerWowApi` call order in `wow-api.ts`:

1. Nil-metatable namespace setup (existing, unchanged)
2. `registerStubs(lua, flavor)` ← new
3. `registerOverrides(lua, flavor)` — `api/` overrides (`C_ScriptedAnimations`, `C_UIColor`, `C_CurveUtil`)
4. Remaining inline stubs (`C_Texture.GetAtlasInfo`, `C_Timer`, etc.) ← last, always win

## Change detection

Two hashes guard writes:

- `@gen-hash` (source `.lua` bytes) — skip rewrite if source unchanged
- `@sig-hash` (function-name:return-type fingerprint) — classic delta detection: if classic sig-hash matches retail sig-hash, skip writing to `classic/` (retail covers it)

Manifests are always fully regenerated.

## Lua parsing

Each `.lua` file is executed in a reused wasmoon sandbox. `APIDocumentation:AddDocumentationTable` (colon call — `self` is first arg) is stubbed as a collector. The collected table is serialized to JSON within Lua via a `_SER` helper (avoiding wasmoon's JS-object-to-userdata conversion), then `JSON.parse`d in TypeScript.

The sandbox also stubs `Constants.*` (needed by some files that set validation constants), `Enum.*`, and a handful of global number constants (`MAX_STABLE_SLOTS` etc.) that certain doc files reference.

Empty Lua tables (`Tables = {}`, `Events = {}`) serialize as `{}` not `[]` — all array fields are normalized with a `toArr()` helper that accepts both forms.

## Generator behavior per flavor run

1. Extract `Interface/AddOns/Blizzard_APIDocumentationGenerated/*.lua` to temp dir via cascTool (reuses `extractPaths` from `src/assets/extract-core.ts`, same pattern as `dev/extract.ts`)
2. Parse all `.lua` files via wasmoon sandbox
3. Build struct name registry (all `Type = "Structure"` table names) for zero-value decisions
4. **Retail run:** write/update `retail/<Namespace>.ts` (hash-skip if unchanged); write `manifest.retail.ts`
5. **Classic/classic_era run:** compare each namespace against `retail/`:
   - Absent from `retail/` → write to `<flavor>/<Namespace>.ts`
   - Present in `retail/` but return shapes differ → write to `<flavor>/<Namespace>.ts`
   - Present in `retail/` and identical → skip
   - Write `manifest.<flavor>.ts`
6. Regenerate `api-stubs/index.ts`

## TypeScript types for parsed data

```typescript
interface DocField {
  Name: string;
  Type: string;
  Nilable: boolean;
  IsArray?: boolean;
  InnerType?: string;
}
interface DocFunction {
  Name: string;
  Arguments?: DocField[];
  Returns?: DocField[];
}
interface DocTable {
  Name: string;
  Type: string;
  Fields?: DocField[];
}
interface DocEvent {
  Name: string;
  LiteralName?: string;
}
interface DocModule {
  Name: string;
  Namespace?: string;
  Type?: string;
  Functions?: DocFunction[];
  Tables?: DocTable[];
  Events?: DocEvent[];
}
```

Note: `InnerType` is populated when `Type = "table"` to identify the element type (e.g. `InnerType = "AtlasInfo"`). The stub always uses `_tbl` in either case; `InnerType` is present for future type-generation use.
